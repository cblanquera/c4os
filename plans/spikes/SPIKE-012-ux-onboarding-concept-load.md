# Objective

Validate whether users can understand the MVP concepts and approval flow without being overwhelmed.

# Context

The UX spec describes a dense command-center interface. Reviews warn that users may not understand projects, sessions, agents, tools, skills, plugins, MCP, models, BYOK, worktrees, artifacts, and approvals.

# Questions To Answer

 - What concepts must appear in MVP onboarding?
 - Can non-essential concepts be hidden until V1 or later?
 - Do users understand project-root scoping?
 - Do users understand approval prompts?
 - How should denied actions and failed tools be explained?
 - What minimum accessibility criteria apply before validation?

# Hypothesis

MVP onboarding should explain only project, session, provider key, approval, tool activity, and changed files. Hiding plugins, MCP, skills, and worktrees reduces confusion.

# Investigation Plan

 - Review MVP user journeys and UX spec.
 - Create concept inventory and classify as MVP, hidden, or excluded.
 - Draft low-fidelity onboarding script or flow for research review.
 - Define approval prompt comprehension checks.
 - Identify accessibility requirements needed for validation.

# Success Criteria

 - MVP concept set is defined.
 - Onboarding comprehension risks are documented.
 - Approval prompt comprehension test questions are drafted.
 - Non-MVP concepts to hide are listed.

# Decisions Unlocked

 - ADR-001: Product Scope And MVP Audience.
 - ADR-021: UX Layout And Interaction Model.
 - MVP user journeys.

# Estimated Effort

2 to 3 product/design research days.
