## Checkpoint Rules

- Checkpoints are implementation sequencing tools, not MVP scope boundaries.
- The MVP remains the full documented/r04 feature set unless explicitly moved
  out later.
- Do not call checkpoint 1 the MVP.
- Do not move documented r04 features to post-MVP through checkpoint language.
- Do not treat security-only progress as product viability.
- Do not treat mocked UI as product-complete.
- Do not freeze implementation before all documented features are planned.

## Checkpoint Strategy

Implementation should start with UI shell first, then backend/runtime
capability.

## Checkpoint 1 Proof

Checkpoint 1 proves:

- folder trust
- provider setup
- OpenCode session
- visible agent response/status
- persisted session

Checkpoint 1 is only a progress review gate. Product MVP completion requires
all documented/r04 features.
